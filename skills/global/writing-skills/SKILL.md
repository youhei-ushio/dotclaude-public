# skill の書き方（SKILL.md 作成規約）

`add-hook` が hook 追加を型化するのと同様に、**skill 作成を型化**する。既存 skill 群に
暗黙的に共有されている規約を明文化し、frontmatter 不備で auto-load されない・トリガー文が
弱くて起動しない、といった事故を防ぐ。

## 最重要原則: description は「説明」ではなく「auto-load トリガー仕様」

skill は frontmatter の `description` を手がかりに auto-load される。description は読み物の
説明文ではなく、**いつこの skill を起動すべきかの判定条件**として書く。トリガー語が弱いと、
必要な場面で起動しない（= 存在しないのと同じ）。

## 手順

### 1. 配置する

- すべての skill は `skills/global/<name>/SKILL.md` に置く（`<name>` は frontmatter の `name` と一致）。
- スタック非依存で常用するものか、特定スタック前提（例: 特定言語・フレームワーク前提）かは、本文の前提記述と
  後述の README 登録先テーブルで区別する（ディレクトリは分けない）。

### 2. frontmatter を書く

```yaml
---
name: <kebab-case の skill 名>        # ディレクトリ名と一致させる
description: |
  <いつ起動すべきかの状況・症状を具体的に>。
  「<トリガー語1>」「<トリガー語2>」「<トリガー語3>」のような自然言語でも起動。
---
```

- `name`: kebab-case。ディレクトリ名と一致させる。
- `description`: **起動条件を具体的に** + **「『…』『…』のような自然言語でも起動」形式で
  トリガー語を複数列挙**。否定文や冗長な説明はトリガー精度を下げるので本文に回す。

### 3. 本文を書く（構成）

既存 skill が共通して持つ流れに沿う:

1. **目的**（なぜこの skill があるか / 防ぎたい失敗）。
2. **制約・絶対ルール**（あれば。「短縮禁止」のように、判断を主観でなく客観条件で固定する）。
3. **手順**（番号付き。分岐は「機械的に判定できる条件」で書き、読み手の解釈に委ねない）。
4. **注意事項**（副作用、前後の skill との関係、棲み分け）。

- 既存 skill・組み込み skill への参照は**実在するものだけ**にする（dead link を作らない。
  特に公開テンプレートでは、リポに無い skill 名を「接続先」として書かない。
  同梱・中立な skill（常に一緒に配布されるもの）は直接参照し、「配布先にあれば」guard は
  任意配布の skill（特定スタック前提など、配布先で opt-out されうるもの）の参照にのみ付ける）。

### 4. 登録する

- **README**: スタック非依存なら「フレームワーク非依存」テーブル、特定スタック前提なら
  「特定スタック前提」テーブルに 1 行追記する。
- **settings.json**: 必要なら `permissions.allow` に `Skill(<name>)` を追記する
  （skill 実行時の確認を省く allowlist。auto-load 自体は frontmatter で完結する）。

### 5. 検証する

- frontmatter が妥当な YAML か（`name` / `description`）。
- description が起動条件として機能するトリガー文か（単なる説明になっていないか）。
- 本文の参照先 skill / パスが実在するか（dead link が無いか）。
- README / settings.json への登録漏れが無いか。
- 新しい規約・方針の追加（仕様判断）にあたるなら、配布先に ADR 運用があれば
  `documentation-standards` の ADR 規約に従って ADR を起票する。

## 注意事項

- skill は「字面の遵守＝精神の遵守」になるように書く。手順を機械的になぞれば意図が満たされる
  状態を目指し、解釈の余地で結果がぶれないようにする。
- 既存 skill と役割が重なるなら、棲み分け（どちらをいつ使うか）を本文に明記する。
- hook を伴う skill（特定コマンドを skill へ誘導したい等）は `add-hook` も併せて使う。
