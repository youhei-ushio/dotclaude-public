# レビューチェックリスト

マニュアル本体を書き終えた後、または「マニュアルレビューして」と指示された場合に
上から順に確認する。

## 🔴 重大 (現場が辿れない / 配布前に必須)

### 1. メニュー導線が実装と一致しているか

- [ ] 大カテゴリ名 (例: 「受注管理」「在庫管理」) がナビゲーション実装と一致
- [ ] サブメニュー名 (例: 「注文作成」) が実装と一致
- [ ] PC ナビとスマホナビ (レスポンシブ) の両方を確認し、提供されている場所を正確に書く
- [ ] PC ナビにメニューがない機能は「スマホ専用」と明示する
- [ ] スマホ専用機能を「URL を直接ブラウザに入力」と案内するのは NG。**メニュー導線を必ず書く**

### 2. ボタン名・リンク名が実装と一致しているか

- [ ] 「次へ」「作成」のような **一般語をマニュアルが勝手に使っていないか**
- [ ] 実装のテンプレート（Laravel Blade / React JSX / Vue SFC 等）の
      ボタンラベル・i18n キーを確認して引用する
- [ ] 押下後の画面遷移 (画面名・URL) も整合確認

### 3. 操作手順の連続性

- [ ] 「画面 A から画面 B への遷移」がどのアクション (チェック / ボタン / リンク) で起きるか書かれている
- [ ] 中間画面 (一覧画面、確認画面) を飛ばしていない
- [ ] セッション状態 (例: 一覧で選択した ID をセッション渡し) が前提になっている遷移を見落としていない

### 4. 物理名・開発用語が残っていない

`TERMINOLOGY.md`「チェック方法 (grep)」セクションの **4 種類すべて** を実行し、
それぞれヒット 0 件であること。

- [ ] **#1 固定 NG ワード** grep (テーブル名 / カラム名 / パス / Issue 参照等)
- [ ] **#2 「業務語（英名）」の括弧併記** grep
      例: `出荷梱包（Fill）画面` → NG。`出荷梱包画面` に直す。
      正規表現: `[ぁ-んァ-ヶ一-龯]+（[A-Z][A-Za-z]+）`
- [ ] **#3 プロジェクトの UI コンポーネント / Model クラス名動的抽出** grep
      例: `OrderForm`, `OrderModal`, `Fill` 等。自フレームワークのコンポーネント
      基底クラスを継承するクラス全てをパターン化する。
- [ ] **#4 ファイル名・パスの混入** grep
      例: `OrderForm.blade.php`, `app/Contexts/...`, `.php`/`.tsx` 等の拡張子

> **重要**: #1 の固定 NG リストだけだとプロジェクト固有のコンポーネント
> 名漏れ（特に業務語に擬態した `Fill` のような短い英名）を捕捉できない。
> **4 種類すべて実行する** こと。

### 5. 開発リファレンス末尾セクションがない

- [ ] 「関連 Issue / ADR」セクションが残っていないか (削除対象)
- [ ] 「Closes #XXXX」「本マニュアルの対応 Issue」のような PR 文脈語が残っていないか

### 5.5. 実装との整合スポットチェック (事実誤認の検出)

`SKILL.md` の Step 6.5 と同じ観点。grep では捕捉できない **事実誤認**
(動線間違い / ボタンラベル違い / バリデーション制約の食い違い等) を
実装ファイルと突合せて確認する。Plan / Explore エージェント等の subagent
に実装読みを任せると効率的。

- [ ] メニュー導線 (大カテゴリ・サブメニュー名) がナビゲーション実装と一致
- [ ] ボタン / リンク名が i18n キー or 直書きラベルと完全一致
- [ ] 「○○ボタンを押すと△△画面が開く」が実装のディスパッチ / ルーティングと一致
  - 特に **類似ボタンが別画面を開くケース** (例: 「出荷」と「在庫予約」が
    別モーダル) を混同していないか
- [ ] チェックボックス / トグルの ON/OFF 挙動が双方向バインディング
      （Livewire `wire:model` / Vue `v-model` / React state 等）と一致
- [ ] バリデーション制限 (文字数・数値範囲・必須) がフォームバリデーション実装と一致
- [ ] 保存ロジック (NULL 化条件・正規化処理) がコンポーネント / コントローラーの実装と一致
- [ ] 印字物 (ラベル / 帳票) の項目・条件付き表示がテンプレートの分岐と一致
- [ ] 状態判定 (バナー表示条件・エラー時抑制等) がテンプレートの条件式と一致
- [ ] **動的トリガーのデータフロー追跡** — 表示条件で参照される値 (例: 「ある集計値が
      閾値を超えたら警告バナーを表示」のような分岐) が「業務フロー上のどの
      タイミングで作成され、どこから条件を満たす値になるか」を追う。
      テンプレートの条件式と一致しているだけでは不十分。
      - 集計の元テーブルがいつ INSERT されるかを実装パスで確認
      - 業務担当者が見るシナリオ (例: 「特定操作直後に帳票を印刷」)
        での実際のデータ状態を **シナリオシミュレーション** で追跡
      - マニュアル記述「○○すると△△が出る」の **△△が出るタイミング** が
        現場期待と一致するか確認

> grep でクリーンでも、ここで実装と突合せないと「ボタン名違い」「動線違い」
> のような事実誤認は通り抜ける。実運用で複数の事実誤認（ボタンの動線取り違え、
> 「指示」→「登録」のような文言ずれ、連続空白正規化の副作用、表示タイミングの
> 認識違いなど）が捕捉漏れになった教訓を踏まえ、**マージ前に必ずこのスポット
> チェックを実施する**。
>
> 特に「動的トリガーのデータフロー追跡」はテンプレートの条件式を読むだけでは
> 見えない。Plan エージェントに「**この値が業務フロー上いつ条件を満たすか
> をデータ書き込みパスから追跡せよ**」と明示的に指示すること。

## 🟡 中 (情報不足・誤解を招く)

### 6. アクセス権限の表現

- [ ] `is_admin = 1` のような物理フラグではなく「**管理者権限が必要**」のような業務語になっている
- [ ] PC / スマホそれぞれの操作場所が表記されている

### 7. 画面に出る数値・状態の説明

- [ ] 「数量」「箱数」「在庫」など、画面上で表示される値の意味が説明されている
- [ ] 入力欄の「合計値」と「明細値」の関係性が明確 (例: 「移送数量 = 各箱の合計」)
- [ ] 単位 (個、枚、箱) が一貫している

### 8. エラー時の挙動

- [ ] バリデーションエラーが出る条件 (例: 合計不一致) が書かれている
- [ ] 入力範囲制約 (例: 1 以上, 99 以下) が「現場が直感する範囲」で書かれている

### 9. 印刷物 (ラベル / 帳票) の項目説明

- [ ] 印字項目を **表示位置 (上段/中段/下段) ごとに表で整理**
- [ ] 項目名・例・説明の 3 列構成が望ましい
- [ ] スクショと項目表の順序が一致 (上から下の物理位置で並べる)
- [ ] 条件付き表示 (例: 1 箱のときは箱情報を出さない) を補足

### 10. 業務扱いの説明 (数量乖離・拒否・差分)

- [ ] 数量差や拒否があったとき、システム上のデータがどう変わるか
- [ ] 現場の次の行動 (検査工程に回す / 担当部署に差し戻す等) が明示されている

## 🟢 低 (補強)

### 11. よくある質問 (Q&A)

- [ ] 現場が直感的に抱きそうな疑問が 5〜7 件カバーされている
- [ ] 「画面上限値」「やり直しの可否」「ログ確認方法」「不良時の流れ」は最低でも含める
- [ ] 回答は「現場が次に取る行動」中心 (システム担当に依頼 等)

### 12. スクショの質

- [ ] 撮影画面に開発用語 (Issue 番号、デバッグ表示) が映り込んでいない
- [ ] スクショの alt テキストが業務語で書かれている
- [ ] 数量等の例が業務上それらしい値 (50/50/47 のような不自然な値より、現場で起きやすい配分)

### 13. 用語一貫性

- [ ] 同じ概念に複数の呼び方が混在していない (例: 「明細」「行」「項目」が混在)
- [ ] 「箱」「箱情報」「箱番号」「総箱数」の使い分けが一貫
- [ ] 数式記号 (`=`, `<`, `>`) より自然な日本語表現 (「以上」「以下」「と一致」)

### 14. 平易さ (現場語)

- [ ] 「再配分」「N 箱」「処理者」のような硬い・抽象的な語を避ける
- [ ] 「振り直し」「正しい箱数」「ラベルを発行した担当者」のような具体語

### 15. 別フローへの言及

- [ ] 他の操作 (「戻り受領」等) に触れる場合、「本マニュアルの範囲外」「別画面の操作」と注記
- [ ] 別マニュアルが存在する場合はリンク (パスで参照)

## ✅ 良い兆候 (チェック完了の目安)

- 開発者が読まなくても現場が手順を辿れる
- スクショと本文の遷移が一致している
- 印刷物の項目表が「物理位置 (上段/中段/下段)」で構造化されている
- 拒否・乖離・誤受領などの **業務扱い** が業務語のみで説明されている
- ログ確認は「システム担当に依頼」になっている

## 提示形式

レビュー結果をユーザーに返すときは、上記の優先度 (🔴 / 🟡 / 🟢) と通し番号で
整理する。各指摘について:

```
### Rn. 観点タイトル

[元の記述]

→ [推奨修正]
```

の形式で提示すると、ユーザーが修正範囲を一目で把握できる。
