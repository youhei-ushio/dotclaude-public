# Issue対応フロー

## 概要

GitHub Issue の対応を一括で実行します。mainブランチの最新取得からPRレビューまでを自動的に行います。

本 skill は **フレームワーク非依存** です。本文中に登場するコマンド・パス・
テストランナーの具体例（Laravel + Sail / npm / dotnet / cargo 等）は一例に
過ぎないので、自プロジェクトのスタックに合わせて読み替えてください。
ドキュメント配置ルール・MUSTルールも convention の一例なので、必要に応じて
カスタマイズします。

## 実行フロー

### 1. 最新mainの取得

- `main` ブランチに切り替え
- `git pull` で最新を取得

### 2. Issue内容の確認

- `gh issue view <issue番号>` でIssueの内容を取得
- タイトル、本文、ラベル、コメントを確認
- 対応方針を把握

### 3. 作業ブランチ作成

- `feature/issue-<issue番号>-<概要>` 形式でブランチを作成
- ブランチ名の概要部分はIssueタイトルから英語のケバブケースで生成

### 4. ADR作成

- 仕様の追加・変更を伴う場合は、必ずADR（Architecture Decision Record）を作成する
- 仕様変更を伴わない純粋なバグ修正・リファクタリングではADR不要
- **配置先・ファイル名規約は `/documentation-standards` skill 参照**（ADR の連番採番ルールも同 skill 側で定義）
- ADRテンプレート（workflow 固有なので本 skill に保持）:
  ```markdown
  # <Issue番号>. タイトル

  ## ステータス
  承認済み

  ## コンテキスト
  どのような課題・背景があるか。

  ## 決定
  何をどう変更するか。

  ## 理由
  なぜこの決定を選んだか。代替案との比較。

  ## 影響
  この決定によって生じる影響・トレードオフ。
  ```

### 5. 設計書チェック（実装前ゲート）

設計書の配置先（`docs/specifications/` 系）と命名規約は `/documentation-standards`
skill 参照。本ステップでは workflow としての **ゲートの掛け方** だけ定義する。

- 対応する設計書が既に存在するか確認する
- **設計書が存在しない場合:**
  1. 設計書を作成する
  2. 設計書のみのPRを作成する（実装PRとは別ブランチ・別PR）
  3. **⛔ ここでフローを中断する。ユーザーに「設計書PR #XX を作成しました。承認後に実装を開始します」と報告し、指示を待つ**
  4. ユーザーから実装開始の指示があったら、実装ブランチに切り替えてステップ6（実装）に進む
- **設計書が既に存在する場合:** そのままステップ6（実装）に進む

**設計書を新規作成する場合、可能な限り図（drawio → SVG）を含めること**。
文章だけの仕様書は認知負荷が高い。図で視覚的に表現することで、人間が素早く
全体像を把握できるようにする。

仕様の内容に応じて適切な図の種類を選択する:

| 仕様の内容 | 図の種類 | 命名パターン |
|-----------|---------|------------|
| テーブル設計・リレーション | ER図 | `<機能名>-er.drawio` |
| 処理フロー・業務フロー | フローチャート | `<機能名>-flow.drawio` |
| ステータス遷移 | 状態遷移図 | `<機能名>-state.drawio` |
| システム間連携・API呼び出し | シーケンス図 | `<機能名>-sequence.drawio` |
| 判定ロジック・分岐条件 | デシジョンフロー | `<機能名>-decision.drawio` |
| データの流れ・変換処理 | データフロー図 | `<機能名>-dataflow.drawio` |

1つの設計書に複数種の図を含めてよい（例: ER図 + フローチャート）。
**drawio / SVG ファイルの配置先は `/documentation-standards` 参照**。

drawioの色規約（既存図との統一）:

| 色 | 用途 |
|----|------|
| 青 `#dae8fc` | ユーザー・認証系エンティティ、入力ステップ |
| 緑 `#d5e8d4` | マスタデータ、正常系、完了状態 |
| 黄 `#fff2cc` | 中間テーブル、待ち状態、注意 |
| 紫 `#e1d5e7` | カテゴリ・処理エンティティ、中間処理 |
| 赤 `#f8cecc` | トランザクション、アラート、エラー、キャンセル |
| グレー `#f5f5f5` | 初期状態、対象外、補足情報 |

### 6. 実装

- Issueの要件に基づいて実装を行う
- 必要に応じてテストを追加・更新
- 実装完了後、全テストを実行して通過を確認:
  ```bash
  # 例（自プロジェクトのテストコマンドに置換）:
  # Jest/Vitest:     npm test
  # .NET:            dotnet test
  # Go:              go test ./...
  # Cargo:           cargo test
  # Pytest:          pytest
  # Rspec:           bundle exec rspec
  # Laravel + Sail:  docker exec <container> php artisan test
  # PHPUnit (Sail 非使用時): ./vendor/bin/phpunit  ※Sail 環境ではコンテナ内で実行する
  ```
- **成果物がドキュメントのみ（設計資料、仕様書等）でコード変更がない場合、テスト実行・ブラウザテストケース追加・ブラウザテスト実行（ステップ11・12）はスキップする**
- **設計資料にフロー図や構成図を含める場合は、drawio形式で作成し、SVGにエクスポートしてMarkdownに埋め込む**
  - drawio / SVG の配置先は `/documentation-standards` 参照
  - **SVG への変換は `~/.claude/hooks/run-drawio-export.sh` を使う**。`Bash` で `drawio --export` / `Xvfb` を直接呼ばない
  - SVGファイルもdrawioと同じディレクトリに配置する
  - Markdownからは `![図の説明](diagrams/ファイル名.svg)` で参照する

### 7. コミット・プッシュ

- 変更をコミット（適切なprefixを使用: feat, fix, refactor, test, docs）
- コミットメッセージは変更内容を端的に説明
- リモートにプッシュ

### 8. PR作成

- **`/create-pr` スキルの手順に従ってPRを作成する**
- Issue対応時の追加ルール（フォーマット、成果物リンク、印刷イメージ等）も `/create-pr` に定義済み

### 9. PRレビュー

- 作成したPRに対してコードレビューを実施
- レビュー観点（自プロジェクトのスタックに合わせて拡張する）:
  - テストの追加・更新が適切か
  - セキュリティ上の問題がないか
  - パフォーマンスへの影響がないか
  - フレームワークのベストプラクティスに沿っているか
    （Laravel + Livewire / Rails / Django / React 等それぞれ）
  - N+1クエリ問題がないか（ORM 利用時）
- 問題があれば修正してコミット・プッシュ
- レビューの改善提案に対応した場合は、対応内容をPRコメントとして投稿する（どの提案に対してどう対応したかを明記）

### 10. マニュアルレビュー

- 対応内容が現場向けマニュアルに反映すべきかを判断（配置先は `/documentation-standards` 参照）
- 確認観点:
  - 新機能や動作変更がユーザー向けマニュアルに記載されているか
  - ステータス遷移やフローの変更が反映されているか
  - 既存の説明と矛盾する記述がないか
- 反映漏れがあればマニュアルを更新してコミット・プッシュ
- マニュアル新規作成・大幅改訂は `/create-manual` skill を使う

### 11. ブラウザテストケース追加

- 対応内容に応じたブラウザテストケースを `tests/browser-test-cases/` に追加
- 既存のテストケース（TC-XX）の採番に続ける形で追加
- テストケースのフォーマット:
  ```
  ### TC-XX: テストケースタイトル
  | 項目 | 内容 |
  |------|------|
  | 前提 | 前提条件 |
  | 画面 | 操作対象の画面パス |
  | 操作 | 実施する操作手順 |
  | 期待結果 | 期待される結果 |
  | 確認ポイント | 重点的に確認する項目 |
  ```
- テストケース追加後にコミット・プッシュ
- **追加したテストケースの番号と概略をPRコメントに投稿する**（MUSTルール参照）

### 12. ブラウザテスト実行

- 追加したテストケースをブラウザで実行（`/playwright-error-detection` skill を併用）
- ログイン情報:
  - プロジェクトのルールを参照すること
  - パスワードを勝手に書き換えないこと
- テストデータは自プロジェクトの REPL（Laravel Tinker / Rails console / IRB / Python shell 等）で作成し、テスト後にクリーンアップする
- テスト結果（PASS/FAIL）を報告する

### 13. Test plan完了

- PRのbodyにTest planのチェックリストがある場合は、ブラウザテスト実行後に全項目を完了させる
- `gh pr edit <PR番号> --body "<更新後の本文>"` でチェックボックスを `[x]` に更新
- 全項目が `[x]` になっていることを確認

### 14. 完了通知（任意）

プロジェクトでチーム通知が必要な場合は、Slack / Teams / Discord webhook 等
への通知 hook を別途用意して呼び出す。本 skill には汎用 hook は同梱しないので、
必要なら自プロジェクトに `slack-notify.sh` 等を追加する。

メッセージフォーマット例:
```
PR #<PR番号>（<PRタイトル>）のtest planを完了しました。

ブラウザテスト結果:
:white_check_mark: <テストケース1の概要>
:white_check_mark: <テストケース2の概要>

全テスト項目がパスしています。
```

テストが FAIL の場合は `:x:` を使用する。

## 注意事項

- 全テストが通過するまでコミットしない
- PRのbodyには必ず `Closes #<issue番号>` を記載する
- 実装に不明点がある場合はユーザーに確認する
- マニュアルへの反映が必要な場合は合わせて更新する（配置先は `/documentation-standards` 参照）
- テスト実行は効率的に行う:
  - 開発中は `--filter` 系オプションで関連テストのみ実行する
  - 全テストはコミット前にのみ実行する
- **コードベースの調査には可能な限りSerena MCPツールを使用する**:
  - `find_symbol` / `get_symbols_overview` でモデル・クラス構造を把握する
  - `find_referencing_symbols` でシンボルの参照箇所を特定する
  - `search_for_pattern` でコードベース横断のパターン検索を行う
  - `list_dir` / `find_file` でファイル探索を行う
  - ファイル全体の読み込みは最小限にし、シンボル単位で必要な情報のみ取得する

## MUSTルール（必須遵守事項）

下記は **issue 対応 workflow に固有のルール**。ドキュメントの配置先・命名規約・
ディレクトリツリー等の汎用ルールは `/documentation-standards` 側で定義されて
いるため、本セクションは workflow の必須事項に絞っている。

- **【ADR必須】** 仕様の追加・変更を伴うIssueでは、必ずADRを作成すること
  - ADRは実装コミットと同一PRに含める
  - 仕様変更を伴わない純粋なバグ修正・リファクタリングではADR不要
  - 配置先・ファイル名規約は `/documentation-standards` 参照
- **【設計書必須】** 対応する設計書が存在しないIssueは、実装前に必ず設計書のPRを作成すること
  - 設計書PRは実装PRとは別に作成する
  - 設計書のレビュー・承認後に実装PRを作成する
  - **設計書PRの承認前に実装を開始してはならない（ステップ5でフローを必ず中断すること）**
- **【ブラウザテスト設計同時作成必須】** 設計書を作成する場合は、同一PRにブラウザテストケースの設計も含めること
  - テストケースは `tests/browser-test-cases/` に追加する（プロジェクトの配置慣習に従う）
  - 設計書の要件・画面仕様に基づいて、実装後に実施すべきブラウザテストケースを事前に設計する
  - テストケースのフォーマットはステップ11に準ずる
  - 設計書PRのbodyに追加したテストケースの番号と概略を記載する
  - 実装PR（ステップ11）では、設計書PRで追加済みのテストケースを再利用し、必要に応じて追加・修正する
- **【ドキュメントリンク必須】** 成果物にドキュメント（設計資料、仕様書、ADR等のMarkdownファイル）が含まれる場合、PRのbodyに必ず「成果物リンク」セクションを設け、GitHub上で閲覧可能なリンクを記載すること
  - ADR・設計書のリンクは必ず含めること
  - リンク形式: `https://github.com/<owner>/<repo>/blob/<branch>/<path>`
  - drawioやSVGファイルはMarkdownに埋め込まれているためリンク不要
  - リンク漏れは絶対に許容しない
- **【ブラウザテストケース記録必須】** ブラウザテストケースを追加した場合、PRコメントに追加したテストケースの番号と概略を必ず投稿すること
  - フォーマット例:
    ```
    ## 追加されたブラウザテストケース
    | テストケース | 概要 |
    |-------------|------|
    | **TC-XX** | テストケースの概要説明 |
    ```
  - テストケースの記録漏れは絶対に許容しない
- **【設計書ダイアグラム必須】** 設計書を新規作成する場合、必ず1つ以上の図（drawio → SVG）を含めること
  - 文章のみの設計書は認知負荷が高いため許容しない
  - 図の種類は仕様内容に応じて選択する（ER図、フローチャート、状態遷移図、シーケンス図、デシジョンフロー等）
  - 設計書Markdownの概要セクション直後に `![図の説明](diagrams/<ファイル名>.svg)` で埋め込む
  - 図の内容は設計書の文章と正確に一致させること（ラベル、ステータス名、処理順序等の不一致は不可）
  - 既存の図と色規約を統一すること（青=ユーザー系、緑=マスタ/正常系、黄=中間/待ち、紫=処理系、赤=エラー/キャンセル、グレー=初期/対象外）
  - drawio / SVG の配置先は `/documentation-standards` 参照
- **【ユーザー指示のissue反映必須】** issue対応中にユーザーから受けた追加指示・方針変更は、該当issueのコメントまたは本文に必ず反映すること
  - 対応中の会話で決まった仕様・方針はissueにも記録する
  - issueを見れば対応経緯がわかる状態にすること
  - 反映漏れは絶対に許容しない
